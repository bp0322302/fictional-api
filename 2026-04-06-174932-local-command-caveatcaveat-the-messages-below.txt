❯ # API For Software Testing MSc Module                                         
                                                                                
## Overview                                                                     
                                                                                
A fictional company needs to solve their fictional problem by having a          
fictional API created.  This fictional company uses Github as the location to   
store its code repositories and run CI/CD workflows.  Employees are allowed to  
use their own Github accounts, and add them to the fictional companies Github   
org.  It is difficult however for employees without admin access to Github to   
determine which employee an account relates to if they use a nickname.          
                                                                                
This API would provide employees the ability to identify which github account   
maps to which AD account.                                                       
                                                                                
## Application Requirements                                                     
                                                                                
- For the API, fastapi should be used, with the backend data stored in a        
postgresql database.                                                            
- 3 GET Endpoints, a /repo a /user and a /group endpoint                        
- Query for a user, or users, and find out what groups they are a member of,    
and what repositories they have access to, and the access level.  Show user     
details, such as name and email, Team and Job Title.                            
- Query for a group, or groups, and find out what repositories they have access 
 to, and the access level.  Find out the users who are members of that group.   
Show what AD group the github group relates to.                                 
- Query for a repository, or repositories and see what users and groups have    
access to them, and what access level                                           
- Each endpoint should take at least one variable, which would change the       
information returned from the endpoints                                         
- A fully integrated API is not required right now, instead a prototype with    
mocked data.                                                                    
                                                                                
- A dataset should be mocked up and provided in a script to load into the       
database, with at least 10 rows of fake data for testing.                       
- OpenAPI Spec should be created for each endpoint                              
                                                                                
## Additional Requirements                                                      
                                                                                
- A hypothetical requirements document for this API (see                        
https://en.wikipedia.org/wiki/User_requirements_document)
